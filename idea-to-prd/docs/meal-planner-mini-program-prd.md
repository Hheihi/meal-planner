# 今日吃什么 — 每餐点菜助手小程序

## Document Metadata
- Original PRD date: 2026-03-13
- Latest update date: 2026-03-13
- Update rule: Unmarked content belongs to the original draft. Incremental changes are explicitly labeled with dates.

## 1. 概述

"今日吃什么"是一款微信小程序，帮助自己做饭的用户解决"每餐不知道吃什么"的选择困难问题。小程序提供按菜系分类的菜品库，每道菜附带菜谱和做法说明，支持智能推荐和手动浏览两种选菜方式。用户可以按早/中/晚餐规划每餐菜单，记录每日饮食，并查看轻量级营养摄入信息。同时支持家庭群组功能，家庭成员之间可以共享查看今日菜单。

## 2. 问题陈述

- **现有问题**：很多人每天在"吃什么"这个问题上花费大量时间纠结，尤其是自己做饭的用户，既要考虑吃什么、又要考虑怎么做、还要兼顾营养均衡，决策成本高。
- **影响人群**：在家做饭的个人和家庭用户。
- **为什么现在要做**：市面上的菜谱类产品侧重"怎么做"，缺少从"吃什么"决策到"餐食规划"再到"饮食记录"的完整闭环工具。

## 3. 目标

### 业务目标
- 建立一个有粘性的工具型小程序，通过每日高频使用场景获取稳定用户量
- 积累用户饮食偏好数据，为后续个性化推荐和商业化打基础

### 用户目标
- 快速解决"这顿吃什么"的决策问题
- 获取菜谱和做法，降低做饭门槛
- 记录每日饮食，了解基本营养摄入情况
- 家庭成员之间同步每日菜单，方便沟通和准备

## 4. 非目标

- 本版本不覆盖外卖/外出就餐场景（后续迭代考虑）
- 不做深度营养管理（如结合身高体重的个性化饮食方案）
- 不做用户收藏夹或常做菜品快捷入口（后续考虑）
- 不做菜品UGC（用户上传菜谱）
- 不做社交功能（如分享到朋友圈、看别人吃了什么）
- 不包含付费或商业化功能
- 家庭群组不涉及协作点餐、投票等复杂交互

## 5. 目标用户

### 主要用户
- **家庭做饭者**：每天需要为自己或家人准备餐食的人，典型画像是家庭中负责做饭的成员
- **独居做饭者**：自己做饭但经常不知道吃什么的年轻人或上班族

### 使用场景
- 通常在做饭前 30 分钟到 1 小时打开小程序决定吃什么
- 做完饭或吃完饭后记录本餐饮食

## 6. 核心用户场景

### 场景一：不知道吃什么，寻求推荐
用户打开小程序，首页展示智能推荐的菜品（基于历史饮食记录和营养均衡原则）。用户浏览推荐结果，看中某道菜后查看详情（菜谱、做法、营养信息），确认后将其加入今日某一餐的菜单。

### 场景二：想吃某个菜系，主动浏览
用户进入菜系分类页面，选择一个菜系（如川菜、粤菜），浏览该菜系下的菜品列表，选择感兴趣的菜品查看详情并加入菜单。

### 场景三：规划一整天的餐食
用户分别为早餐、午餐、晚餐选择菜品，形成当天的完整菜单。做完饭后确认记录。

### 场景四：家庭共享菜单
用户A（做饭的人）规划好今日菜单后，家庭群组中的其他成员（如配偶、孩子）可以打开小程序看到今天各餐吃什么。

### 场景五：查看饮食记录和营养信息
用户查看过去一段时间的饮食记录，了解每日/每餐的营养摄入概况（热量、蛋白质、碳水、脂肪）。

### 边界场景
- 用户将推荐的菜品全部跳过：支持刷新推荐或切换到手动浏览
- 当天没做饭 / 外出就餐：允许跳过记录或手动输入简单文字记录
- 家庭成员在菜单确定前查看：显示"菜单规划中"状态

## 7. 方案概述

### 产品架构

小程序包含以下核心模块：

1. **首页（推荐 + 今日菜单）**
   - 顶部展示今日菜单概览（早/中/晚餐，已选菜品）
   - 下方展示智能推荐菜品卡片流
   - 支持一键加入某一餐

2. **菜系浏览**
   - 按菜系分类（如川菜、粤菜、湘菜、鲁菜、江浙菜、西餐、日韩料理等）
   - 每个菜系下展示菜品列表
   - 支持搜索菜品名称

3. **菜品详情页**
   - 菜品图片
   - 食材清单
   - 制作步骤（图文）
   - 营养信息（热量、蛋白质、碳水、脂肪）
   - "加入菜单"按钮（选择加入哪一餐）

4. **今日菜单**
   - 按早/中/晚餐分栏展示已选菜品
   - 支持移除菜品、更换菜品
   - 显示每餐及全天的营养摄入汇总
   - 确认完成后转为饮食记录

5. **饮食记录**
   - 日历视图，按日期查看历史饮食
   - 每日营养摄入汇总
   - 支持查看某一天各餐吃了什么

6. **家庭群组**
   - 通过微信分享链接邀请成员加入家庭
   - 家庭成员可查看今日菜单
   - 群组内仅共享查看，不支持协作编辑

7. **个人中心**
   - 微信授权登录
   - 口味偏好设置（可选，用于优化推荐）
   - 家庭群组管理
   - 饮食记录入口

### 核心交互流程

```
打开小程序
  → 首页：查看智能推荐 / 进入菜系浏览
    → 查看菜品详情
      → 加入某一餐菜单
        → 今日菜单页查看和调整
          → 确认完成 → 自动记录到饮食日记
```

## 8. 范围

### 在范围内（V1）
- 微信小程序，微信登录
- 菜系分类浏览（每个菜系几十道菜品）
- 菜品详情：菜谱、做法步骤、食材清单、基础营养信息
- 智能推荐：基于饮食记录 + 营养均衡
- 按餐（早/中/晚）规划今日菜单
- 饮食记录（日历视图 + 营养摘要）
- 家庭群组：分享链接创建，成员共享查看今日菜单
- 菜品搜索

### 不在范围内（V1）
- 外卖/外出就餐场景
- 深度营养管理与个性化饮食建议
- 用户上传菜谱（UGC）
- 收藏夹 / 常做菜品
- 社交分享功能
- 付费功能 / 商业化
- 家庭群组内协作点餐或投票

## 9. 需求

### 功能性需求

**菜品库**
1. 系统按菜系对菜品进行分类展示，菜系包括但不限于：川菜、粤菜、湘菜、鲁菜、江浙菜、西餐、日韩料理等
2. 每个菜系下初始包含几十道菜品
3. 每道菜品包含以下信息：菜品名称、菜品图片、食材清单、制作步骤（图文）、基础营养信息（热量 kcal、蛋白质 g、碳水化合物 g、脂肪 g）
4. 支持按菜品名称搜索

**智能推荐**
5. 首页展示个性化推荐菜品列表
6. 推荐算法综合考虑两个因素：a) 用户近期饮食记录，避免短期内重复推荐相同菜品；b) 营养均衡原则，倾向推荐能补充用户近期摄入不足的营养素的菜品
7. 用户可刷新推荐列表获取新的推荐
8. 新用户无历史数据时，展示热门菜品或随机推荐

**今日菜单**
9. 用户可将菜品加入早餐、午餐、晚餐三个时段的菜单
10. 每餐可添加多道菜品，无上限限制
11. 用户可从菜单中移除已添加的菜品
12. 今日菜单页面显示每餐的营养小计和全天营养合计
13. 用户确认某一餐完成后，该餐自动记录到饮食日记

**饮食记录**
14. 以日历形式展示历史饮食记录
15. 点击某一天可查看当天各餐的菜品和营养信息
16. 展示每日营养摄入汇总（热量、蛋白质、碳水、脂肪）
17. 当天未通过菜单记录时，支持手动添加简单文字记录

**家庭群组**
18. 用户可创建一个家庭群组
19. 通过生成微信分享链接邀请其他用户加入群组
20. 群组成员可查看群组内任意成员的今日菜单
21. 群组创建者可移除成员
22. 每个用户最多加入一个家庭群组（假设，可根据需求调整）

**用户系统**
23. 支持微信一键授权登录
24. 支持设置口味偏好标签（如偏辣、清淡、素食等），用于辅助推荐（可选设置）

### 非功能性需求

1. **性能**：首页加载时间不超过 2 秒（含推荐列表）
2. **数据安全**：用户饮食数据和家庭群组数据需加密存储，不向第三方共享
3. **兼容性**：支持微信 iOS 和 Android 客户端的小程序运行环境
4. **可扩展性**：菜品库数据结构需支持后续扩展（如增加菜品来源、视频教程链接等字段）
5. **内容合规**：菜品数据需经过审核，确保内容质量和合规性

## 11. 指标与验证

### 成功指标
- **日活跃用户数（DAU）**：上线 1 个月后达到目标值（具体数值待定）
- **每用户日均使用次数**：目标 ≥ 1.5 次/天（覆盖午餐+晚餐场景）
- **菜单完成率**：用户添加菜品到菜单后实际确认完成的比例 ≥ 60%
- **次日留存率**：≥ 30%

### 护栏指标
- **推荐跳过率**：用户连续跳过推荐菜品的比例不应持续上升，若上升说明推荐质量下降
- **家庭群组活跃率**：创建群组后 7 天内至少有 1 位成员查看过菜单

### 验证方式
- 通过小程序数据分析后台追踪上述指标
- 上线 2 周后进行首批用户访谈，重点了解推荐满意度和使用障碍

## 12. 依赖与约束

- **微信小程序平台**：需遵守微信小程序开发规范和审核要求
- **菜品数据**：初始数据通过网络爬取获得（小红书、抖音等平台），存在合规风险（详见风险部分），需在上线前完成初始菜品库的建设和审核
- **营养数据**：营养数据为估算值，来源于公开的食物营养数据库，非精准计量（需在产品中对用户做适当提示）
- **服务端基础设施**：需要后端服务支撑用户系统、菜品库、推荐引擎、饮食记录存储

## 13. 风险与待定事项

### 已知风险

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| 菜品数据爬取的合规/版权风险 | 可能面临平台投诉或法律风险 | V1 用爬取数据冷启动，同步探索合规内容来源（如合作菜谱网站、公开食谱数据库），中期过渡到合规数据 |
| 营养数据不准确 | 用户对营养追踪的信任度下降 | 在界面上标注"营养信息仅供参考"，使用权威公开数据库作为数据源 |
| 推荐算法冷启动 | 新用户无数据时推荐效果差 | 新用户阶段使用热门菜品推荐 + 引导设置口味偏好 |
| 家庭群组隐私 | 用户可能不希望饮食被他人看到 | 加入群组需用户主动操作，后续可增加隐私控制选项 |

### 待定事项
1. 每个用户是否只能加入一个家庭群组，还是可以加入多个？（当前假设：一个）
2. 菜品库的具体菜系分类和初始菜品数量需运营侧确认
3. DAU 等成功指标的具体目标数值待业务侧确定
