# 模块三：智能推荐

## 1. 模块概述

智能推荐模块根据用户的历史饮食记录和营养均衡原则，为用户推荐每餐菜品。需处理好新用户冷启动和老用户个性化推荐两种场景。

## 2. 功能清单

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| R-01 | 个性化推荐列表 | P0 | 首页展示推荐菜品 |
| R-02 | 刷新推荐 | P0 | 手动获取新一批推荐 |
| R-03 | 新用户冷启动 | P0 | 无历史数据时的推荐策略 |
| R-04 | 偏好过滤 | P1 | 根据口味偏好和忌口过滤 |

## 3. 推荐算法设计

### 3.1 推荐策略概览

```
用户请求推荐
  │
  ├── 是否为新用户（无饮食记录）？
  │     ├── 是 → 冷启动策略
  │     └── 否 → 个性化推荐策略
  │
  └── 应用偏好过滤 → 输出推荐列表
```

### 3.2 冷启动策略 (R-03)

**触发条件**：用户饮食记录少于 7 天

**推荐逻辑**：
1. **有口味偏好**：优先推荐匹配偏好标签的菜品，其余随机补充
2. **无口味偏好**：按以下混合策略生成推荐列表
   - 50% 热门菜品（全站确认完成次数最高的菜品）
   - 30% 各菜系随机菜品（保证菜系多样性）
   - 20% 纯随机

### 3.3 个性化推荐策略 (R-01)

**推荐逻辑**（按优先级叠加）：

**Step 1 — 排除近期已吃菜品**
- 排除用户近 3 天内已确认完成的菜品
- 近 7 天内的菜品降低推荐权重（不完全排除）

**Step 2 — 营养均衡计算**
- 计算用户近 3 天的平均每日营养摄入：
  ```
  avg_calories = 近3天总热量 / 3
  avg_protein  = 近3天总蛋白质 / 3
  avg_carbs    = 近3天总碳水 / 3
  avg_fat      = 近3天总脂肪 / 3
  ```
- 与参考值对比（成人每日参考）：
  ```
  ref_calories = 2000 kcal
  ref_protein  = 60 g
  ref_carbs    = 250 g
  ref_fat      = 65 g
  ```
- 计算各营养素的偏差率：
  ```
  deficit_ratio = (ref - avg) / ref
  ```
- 对于偏差率 > 20% 的营养素，提升能补充该营养素的菜品权重

**Step 3 — 偏好匹配**
- 匹配用户口味偏好标签的菜品提升权重
- 排除用户忌口标签对应的菜品

**Step 4 — 多样性保证**
- 单次推荐列表中，同一菜系最多出现 3 道菜
- 尽量覆盖不同菜系

**评分公式**（简化版）：
```
score = base_score
      + nutrition_bonus    (营养补充加分，0~30分)
      + preference_bonus   (偏好匹配加分，0~20分)
      - recency_penalty    (近期重复扣分，0~50分)
      + random_factor      (随机因子，0~10分)
```

### 3.4 推荐列表规格

- 每次推荐返回 **10 道菜品**
- 刷新时生成全新列表（不与上一次重复，尽量做到）
- 推荐列表展示为卡片流，可上下滑动浏览

## 5. API 接口

### 5.1 获取推荐列表

```
GET /v1/recommendations?count=10
```

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| count | int | 否 | 推荐数量，默认10 |

**响应**：
```json
{
  "code": 0,
  "data": {
    "list": [
      {
        "id": 101,
        "name": "宫保鸡丁",
        "image_url": "https://...",
        "cuisine_name": "川菜",
        "cooking_time": 25,
        "difficulty": 2,
        "calories": 350,
        "recommend_reason": "营养均衡"
      }
    ],
    "strategy": "personalized"
  }
}
```

`strategy` 字段说明：
- `"cold_start"`: 冷启动推荐
- `"personalized"`: 个性化推荐

`recommend_reason` 可选值：
- `"营养均衡"`: 补充近期不足的营养
- `"口味匹配"`: 匹配口味偏好
- `"热门菜品"`: 冷启动时热门推荐
- `null`: 无特定原因

### 5.2 刷新推荐

```
POST /v1/recommendations/refresh
```

**说明**：请求后端生成新一批推荐（可记录上次推荐的菜品 ID 以避免重复），然后调用获取推荐列表接口。

**请求参数**：
```json
{
  "exclude_dish_ids": [101, 102, 103]
}
```

**响应**：同获取推荐列表。

## 6. 后端实现要点

### 6.1 推荐计算时机

- **方案 A（实时计算）**：每次请求时实时计算推荐结果。适合用户量较小的初期阶段。
- **方案 B（预计算 + 缓存）**：定时（如每小时）为活跃用户预计算推荐列表，存入 Redis。适合用户量增长后。

**V1 建议**：采用方案 A，后续根据性能需求迁移到方案 B。

### 6.2 推荐数据依赖

```
推荐模块需要读取：
├── 菜品表（全量上架菜品）
├── 用户口味偏好（users.taste_preferences）
├── 用户近7天饮食记录（diet_records）
└── 全站菜品热门度统计（可用 Redis 计数器）
```

### 6.3 热门度统计

- 使用 Redis Sorted Set 记录菜品被确认完成的次数
- Key: `dish:popular`
- 每次用户确认完成某道菜 → `ZINCRBY dish:popular 1 <dish_id>`
- 获取热门菜品 → `ZREVRANGE dish:popular 0 49`

## 7. 边界与异常处理

| 场景 | 处理方式 |
|------|---------|
| 菜品库菜品数量过少（<10） | 直接返回全部菜品，不做排除 |
| 用户所有菜品近期都吃过 | 放宽近期排除条件（从3天缩短到1天） |
| 偏好+忌口过滤后候选菜品过少 | 放宽偏好条件，仅保留忌口排除 |
| 推荐接口超时 | 返回预缓存的热门菜品列表作为兜底 |
| 营养数据缺失的菜品 | 不参与营养均衡计算，但仍可被推荐 |
