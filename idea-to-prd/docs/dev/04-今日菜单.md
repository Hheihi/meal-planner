# 模块四：今日菜单

## 1. 模块概述

今日菜单模块是用户每日使用的核心功能，支持按早/中/晚餐规划当天的菜品组合，查看营养汇总，并在完成后生成饮食记录。

## 2. 功能清单

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| M-01 | 查看今日菜单 | P0 | 按早/中/晚餐展示已选菜品 |
| M-02 | 添加菜品到菜单 | P0 | 从推荐/浏览/搜索将菜品加入某餐 |
| M-03 | 从菜单移除菜品 | P0 | 移除已添加的菜品 |
| M-04 | 营养汇总显示 | P0 | 每餐小计 + 全天合计 |
| M-05 | 确认完成某餐 | P0 | 确认后生成饮食记录 |
| M-06 | 首页菜单概览 | P1 | 首页顶部简要展示今日菜单状态 |

## 3. 详细需求

### 3.1 查看今日菜单 (M-01)

**页面结构**：
```
┌─────────────────────────────────┐
│  今日菜单    2026-03-13         │
│                                  │
│  ── 全天营养 ──────────────────  │
│  热量: 1850kcal  蛋白质: 65g    │
│  碳水: 210g      脂肪: 55g     │
│                                  │
│  🌅 早餐                        │
│  ┌─────────────────────────┐   │
│  │ [图] 燕麦粥     280kcal │   │
│  │                    [✕]  │   │
│  └─────────────────────────┘   │
│  小计: 280kcal                  │
│  [+ 添加菜品]  [✓ 确认完成]    │
│                                  │
│  ☀️ 午餐                        │
│  ┌─────────────────────────┐   │
│  │ [图] 宫保鸡丁   350kcal │   │
│  │                    [✕]  │   │
│  └─────────────────────────┘   │
│  ┌─────────────────────────┐   │
│  │ [图] 番茄蛋汤   120kcal │   │
│  │                    [✕]  │   │
│  └─────────────────────────┘   │
│  小计: 470kcal / 蛋白质: 25g   │
│  [+ 添加菜品]  [✓ 确认完成]    │
│                                  │
│  🌙 晚餐                        │
│  (暂无菜品)                      │
│  [+ 添加菜品]                    │
│                                  │
└─────────────────────────────────┘
```

**业务规则**：
- 默认展示当天菜单
- 每餐独立管理，互不影响
- 每餐可添加多道菜品，无数量上限
- 营养数据来源于菜品的 `nutrition` 字段，动态求和

### 3.2 添加菜品到菜单 (M-02)

**触发入口**：
1. 首页推荐卡片 → 点击"加入菜单"
2. 菜品详情页 → 点击底部"加入菜单"按钮
3. 今日菜单页 → 点击"添加菜品"

**交互流程**：
1. 用户点击"加入菜单" → 弹出底部 ActionSheet
2. 选择餐次：早餐 / 午餐 / 晚餐
3. 后端创建 `daily_menus` 记录
4. 显示 Toast"已加入午餐菜单"

**"添加菜品"按钮**（今日菜单页内）：
- 点击后跳转到菜品浏览页面（可选菜系浏览或搜索）
- 选择菜品后返回今日菜单页

**业务规则**：
- 同一菜品不可重复添加到同一餐次 → 提示"该菜品已在午餐菜单中"
- 同一菜品可以出现在不同餐次（如早餐和午餐都吃鸡蛋）

### 3.3 移除菜品 (M-03)

**交互**：
- 菜品条目右侧显示删除按钮 `✕`
- 点击后弹出确认提示"确定从午餐中移除宫保鸡丁？"
- 确认后删除 `daily_menus` 对应记录
- 营养汇总实时更新

**业务规则**：
- 已确认完成的餐次不可移除菜品（需先取消确认）
- 移除后该菜品可重新添加

### 3.4 营养汇总 (M-04)

**计算逻辑**：
```
每餐小计 = 该餐所有菜品的营养值求和
全天合计 = 早餐 + 午餐 + 晚餐 的营养值求和
```

**展示的营养指标**：
- 热量 (kcal)
- 蛋白质 (g)
- 碳水化合物 (g)
- 脂肪 (g)

**业务规则**：
- 菜单变更时实时重新计算
- 营养数据由服务端计算返回，前端也可做本地缓存计算

### 3.5 确认完成 (M-05)

**流程**：
1. 用户做完饭/吃完饭，点击某一餐的"确认完成"按钮
2. 弹出确认提示"确认早餐已完成？完成后将记录到饮食日记"
3. 确认后：
   - 该餐下所有菜品的 `is_confirmed` 设为 `true`
   - 自动为每道菜品在 `diet_records` 表中创建记录（`record_type=1`）
   - 该餐状态变为"已完成"，显示 ✅ 标识
4. 已完成的餐次：
   - 菜品条目显示为已完成状态（可考虑灰色背景）
   - 不可再添加或移除菜品
   - 提供"取消完成"入口（撤销确认）

**取消完成**：
- 仅当天可取消
- 取消后删除对应的 `diet_records` 自动记录
- 菜品恢复为可编辑状态

### 3.6 首页菜单概览 (M-06)

**展示在首页顶部**：
```
┌─────────────────────────────┐
│  今日菜单                    │
│  早餐: 燕麦粥 (✅已完成)    │
│  午餐: 宫保鸡丁、番茄蛋汤  │
│  晚餐: 暂未安排              │
│                  [查看详情 >]│
└─────────────────────────────┘
```

**规则**：
- 每餐最多显示前 2 道菜品名称，超出显示"等N道菜"
- 已确认的餐次显示 ✅
- 未添加菜品的餐次显示"暂未安排"
- 点击"查看详情"跳转到今日菜单页

## 4. API 接口

### 4.1 获取今日菜单

```
GET /v1/menus/today
```

**响应**：
```json
{
  "code": 0,
  "data": {
    "date": "2026-03-13",
    "meals": {
      "breakfast": {
        "dishes": [
          {
            "id": 201,
            "dish_id": 55,
            "name": "燕麦粥",
            "image_url": "https://...",
            "nutrition": {"calories": 280, "protein": 8, "carbs": 45, "fat": 6}
          }
        ],
        "is_confirmed": true,
        "nutrition_subtotal": {"calories": 280, "protein": 8, "carbs": 45, "fat": 6}
      },
      "lunch": {
        "dishes": [...],
        "is_confirmed": false,
        "nutrition_subtotal": {"calories": 470, "protein": 25, "carbs": 35, "fat": 18}
      },
      "dinner": {
        "dishes": [],
        "is_confirmed": false,
        "nutrition_subtotal": {"calories": 0, "protein": 0, "carbs": 0, "fat": 0}
      }
    },
    "nutrition_total": {"calories": 750, "protein": 33, "carbs": 80, "fat": 24}
  }
}
```

### 4.2 添加菜品到菜单

```
POST /v1/menus/dishes
```

**请求参数**：
```json
{
  "dish_id": 101,
  "meal_type": 2,
  "date": "2026-03-13"
}
```

`meal_type`: 1=早餐, 2=午餐, 3=晚餐

**响应**：
```json
{
  "code": 0,
  "data": {
    "menu_item_id": 501
  }
}
```

**错误场景**：
- 该菜品已在此餐次中 → `code: 2002, message: "该菜品已在午餐菜单中"`
- 该餐次已确认完成 → `code: 1003, message: "该餐已确认完成，无法添加"`

### 4.3 从菜单移除菜品

```
DELETE /v1/menus/dishes/:menu_item_id
```

**错误场景**：
- 该餐次已确认完成 → `code: 1003, message: "该餐已确认完成，无法移除"`

### 4.4 确认完成某餐

```
POST /v1/menus/confirm
```

**请求参数**：
```json
{
  "date": "2026-03-13",
  "meal_type": 2
}
```

**业务逻辑**：
1. 将该餐所有 `daily_menus` 记录的 `is_confirmed` 设为 `true`
2. 为每条记录在 `diet_records` 中创建对应条目
3. 更新 Redis 热门度计数

### 4.5 取消确认

```
POST /v1/menus/unconfirm
```

**请求参数**：
```json
{
  "date": "2026-03-13",
  "meal_type": 2
}
```

## 5. 边界与异常处理

| 场景 | 处理方式 |
|------|---------|
| 添加菜品时菜品已下架 | 提示"该菜品已下架，无法添加" |
| 空菜单点击确认完成 | 按钮置灰不可点击，或提示"请先添加菜品" |
| 跨天后查看前一天未确认的菜单 | 保留数据，用户可随时确认或清除 |
| 网络异常导致添加失败 | Toast 提示"添加失败，请重试" |
| 同时操作（如快速连续点击添加） | 前端做防抖处理，后端做幂等校验 |
