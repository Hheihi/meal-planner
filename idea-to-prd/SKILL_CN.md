# 从想法到 PRD

通过两阶段协作循环，帮助用户产出一份高质量的 PRD：

1. 阶段一：访谈并收敛需求
2. 阶段二：用户确认需求就绪后，生成 PRD

核心工作不仅仅是"写一份文档"，而是先减少模糊性，再编写一份反映对话中所做决策的 PRD。

## 默认运行规则

当此技能被调用时，假定用户希望以产品经理式的协作流程进行：

- 如果用户正在描述一个想法、功能、流程变更或产品问题，自动进入阶段一。
- 不要等用户明确说"先问我问题"。
- 仅当用户已提供相当完整的需求，或明确要求立即生成文档时，才可跳过或压缩阶段一。
- 需求充分收敛后，通过快速确认自然过渡到 PRD 生成。

## 适用场景

当用户出现以下情况时使用此技能：

- 说类似"我想做一个功能"、"我有一个产品想法"、"我们想加一个功能"、"帮我定义这个流程"或"我们来想想一个新能力"
- 要求生成 PRD、功能简报、产品规格或需求文档
- 希望你在动笔之前先提问
- 表示想法还很粗糙，希望帮忙完善
- 想把聊天记录、想法碎片或问题陈述转化为结构化的 PRD
- 提到需求发现、需求收敛或产品范围界定（任何语言皆可）

如果用户仅需纯技术架构规格，且未明确要求先产出面向产品的 PRD，则不使用此技能。

## 整体行为

按两个明确阶段运行工作流。

### 阶段一：访谈与收敛

此阶段的目标是帮助用户从模糊想法过渡到稳定的问题陈述、目标范围和可决策的大纲。

阶段一的运行方式：

- 先用自己的话复述想法，让用户快速纠正。
- 将随意的想法陈述视为启动访谈的充分理由。除非意图确实模糊，否则不要询问用户是否想开始阶段一。
- 以小批次提出有针对性的后续问题。每次 1-3 个问题，避免一次性抛出大量问卷。
- 每次用户回复后，总结哪些已确认、哪些仍为假设、哪些仍未解决。
- 推动对话走向决策。产品轮廓已经清晰后，不要继续无休止地发问。
- 如果用户不知道某个答案，提供 2-3 个具体选项及其利弊，让用户选择。
- 区分已确认事实和你的推断，绝不将两者混为一谈。
- 如果用户已提供足够细节，压缩此阶段并快速进入就绪检查。

阶段一需覆盖的领域：

1. 问题与动机
2. 目标用户或角色
3. 核心场景或待完成的工作（Job-to-be-done）
4. 方案形态
5. 范围边界与非目标
6. 约束、依赖与风险
7. 成功指标与上线意图

根据实际情况灵活调整顺序，跟随用户的自然思路，而非强制执行固定脚本。

### 阶段一输出

进入阶段二之前，提供一份简短的收敛摘要，格式如下：

```md
## 当前理解
- 问题：
- 目标用户：
- 主要场景：
- 方案概述：
- 成功信号：

## 假设
- ...

## 待解决问题
1. ...

## 建议的下一步
- ...
```

然后请求明确确认，例如：

"看起来需求已经比较清晰了，可以开始生成 PRD 了。你希望我现在生成草稿，还是再做一轮细化？"

在以下情况之前不要生成最终 PRD：

- 用户明确要求生成，或
- 用户明确表示需求已确定

### 阶段二：生成 PRD

用户确认后，生成一份完整的 Markdown 格式 PRD。

PRD 应具备决策友好性和可实施性，适用于产品与工程团队的交接。表述清晰，避免空话，让利弊权衡可见。

## 文件输出行为

生成 PRD 后，默认保存到当前项目根目录的 `docs/` 目录下。

- 默认文件名：`docs/<feature-slug>-prd.md`
- 从 PRD 标题或功能名称生成 `<feature-slug>`，使用 kebab-case（短横线命名）
- 如果 `docs/` 目录不存在则创建
- 如果用户指定了其他路径，遵循用户指示
- 保存后告知用户确切的文件路径
- 如果没有明确的项目根目录或环境不允许写入，说明问题并将 Markdown 内容以内联方式提供作为备选

创建全新 PRD 时，在文件顶部附近添加文档跟踪元数据：

```md
## 文档元数据
- 初始 PRD 日期：YYYY-MM-DD
- 最新更新日期：YYYY-MM-DD
- 更新规则：未标注的内容属于初始草稿。增量变更以日期明确标注。
```

## 处理 PRD 已存在后的后续需求

默认将后续需求讨论视为对现有 PRD 的更新，而非全新任务。

当用户在 PRD 已存在后带来新信息、范围变更或新需求时：

- 首先，找到并读取现有 PRD（如果路径已知或可从项目的 `docs/` 目录推断）
- 总结当前 PRD 的内容，识别可能受影响的章节
- 仅针对变更部分进行聚焦式迷你发现：变了什么、为什么变、是新增、修改还是超出范围
- 仅在变更相关的地方澄清对用户、流程、需求、验收标准、指标、风险和上线计划的影响
- 除非新信息从根本上改变了产品方向，否则避免重复完整的阶段一访谈

变更充分明确后：

- 就地更新现有 PRD，而非创建第二份独立的 PRD，除非用户明确要求单独文档
- 修改受影响的章节，确保文档内部保持一致
- 尽可能保留原有结构和内容，而非从零重写
- 如果尚无 `## 变更日志` 章节，则在文档末尾添加
- 追加一条带日期的条目总结变更，例如：`- 2026-03-13：新增定时角色切换支持，更新了权限需求。`
- 更新 `## 文档元数据` 中的 `最新更新日期` 字段
- `初始 PRD 日期` 保持不变，以便基线版本可追溯
- 对每个增量新增或修改进行行内日期标注，使读者可区分原始内容和后续更新
- 推荐的行内标注方式：
  - 新增小节或段落：以 `> 增量更新 (YYYY-MM-DD)：<变更内容>` 开头
  - 新增条目或需求：末尾追加 `[新增 YYYY-MM-DD]`
  - 修改条目或需求：末尾追加 `[更新 YYYY-MM-DD]`
- 不要对未触碰的基线内容逐行追加标注；依靠文档元数据规则（未标注内容属于初始草稿）即可
- 告知用户哪些章节被更新了

如果新请求实际上是一个独立功能而非修改：

- 说明它看起来是一个新 PRD 的候选
- 简要解释原因
- 提议创建独立 PRD 或合并到现有文档（如用户希望合并范围）

## 写作原则

- 跟随用户的语言。如果对话使用中文，PRD 也用中文，除非用户另有要求。
- 以对话内容为事实来源。
- 当某些信息仍不完全确定时，明确标注为假设。
- 优先使用具体需求，而非模糊的愿景式表述。
- 当边界情况和运营考量实质性影响范围时，将其纳入。
- 不要假装未解决的问题已经解决。

## PRD 必需结构

除非用户要求使用其他模板，否则使用以下结构：

```md
# <功能名称>

## 文档元数据
- 初始 PRD 日期：YYYY-MM-DD
- 最新更新日期：YYYY-MM-DD
- 更新规则：未标注的内容属于初始草稿。增量变更以日期明确标注。

## 1. 概述
- 一段话概述功能及其重要性。

## 2. 问题陈述
- 当前存在什么问题
- 谁受到影响
- 为什么现在重要

## 3. 目标
- 业务目标
- 用户目标

## 4. 非目标
- 本 PRD 有意不涵盖的内容

## 5. 目标用户
- 主要角色或用户类型
- 相关上下文或工作流

## 6. 核心用户场景
- 主要用例
- 关键边界情况

## 7. 方案概述
- 建议的产品行为
- UX 或流程摘要

## 8. 范围
### 范围内
- ...

### 范围外
- ...

## 9. 需求
### 功能性需求
1. ...

### 非功能性需求
1. ...

## 10. 验收标准
- 使用清单条目或 Given/When/Then 格式（视需要）

## 11. 指标与验证
- 成功指标
- 护栏指标
- 如何判断功能是否成功

## 12. 依赖与约束
- 涉及的团队、系统、策略、时间安排或外部阻碍

## 13. 风险与待解决问题
- 已知风险
- 尚未解决的未知事项

## 14. 上线计划
- 建议的发布方式（如已知）

## 15. 变更日志
- YYYY-MM-DD：创建初始草稿。
```

## 交互准则

- 保持协作态度，而非审问式。
- 如果用户给出零散笔记，先整合再提下一个问题。
- 如果存在多个可能的产品方向，简要对比并推荐一个。
- 如果用户要求立即生成 PRD，先做一次轻量级就绪检查。仅询问最少的高价值澄清问题。
- 如果用户的初始输入只是一个粗略想法，不要直接进入文档模式，自动启动发现模式。
- 如果用户在 PRD 已存在后带来新需求，从变更分析模式开始，而非全新的从零发现。
- 如果用户在阶段二之后要求修改，直接更新 PRD，仅当修改请求揭示了真正的模糊性时才重新开启发现。

## 用户不确定时的合理默认值

当用户不知道某些重要信息时，提出一个合理的占位值并标记为假设。常见示例：

- 目标用户群
- 上线范围
- 成功指标
- 非目标
- 验收标准的措辞

优先保持进度并明确标注假设，而非停滞不前。

## 此技能应处理的示例提示

**示例 1**

输入："我想做一个批量调价工具，给运营经理用的。"

行为：自动进入阶段一，复述想法，围绕用户、流程、权限、安全防护和成功标准提问，需求稳定后过渡到 PRD 生成并将结果保存到 `docs/` 下。

**示例 2**

输入："把这个粗略的功能想法转成 PRD，但在你觉得范围稳定之前先问我问题。"

行为：执行两阶段工作流，仅在明确的就绪检查后才编写 PRD，并将最终文档保存到 `docs/` 下。

**示例 3**

输入："我们想让客服人员能查看客户的完整对话历史。帮我先梳理一下需求。"

行为：自动启动阶段一，收敛问题、利益相关者、权限、边界情况和约束，然后询问是否生成 PRD 并保存到 `docs/` 下。

**示例 4**

输入："基于我们上面讨论的内容，把它转成 PRD，未解决的地方标记为假设。"

行为：跳过阶段一的大部分内容，做一个快速的就绪摘要，然后生成带有标注假设的 PRD 并保存到 `docs/` 下。
